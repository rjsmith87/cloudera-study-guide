The Cloudera stack, in Salesforce terms

Every tool mapped to something you already know. Pattern recognition, not memorization.

Ingestion — getting data in
NiFiLearn first

Visual drag-and-drop data routing. You draw a pipeline on a canvas — "grab this file from an FTP server, transform the JSON, push it to Kafka." No code. All config.

SF equivalent: Flow Builder. You drag processors onto a canvas and connect them, just like Flow elements. Each processor does one thing — route, filter, transform, deliver. If you can build a Screen Flow, you can understand NiFi.
KafkaLearn first

Real-time message bus. Systems publish events to "topics" and other systems subscribe to read them. Nothing gets lost. Everything is ordered. Massively scalable.

SF equivalent: Platform Events. When you publish a Platform Event, any subscriber (Flow, Trigger, external system) picks it up. Kafka is the same concept but handling millions of events per second across an entire enterprise.
Stream processing — reacting in real time
FlinkKnow the concept

Processes data the instant it arrives in Kafka. Filters, aggregates, enriches, and routes events in real time — no waiting for a batch job.

SF equivalent: Platform Event-Triggered Flow. Event fires → Flow runs immediately. Flink is the same idea at massive scale with stateful logic (e.g., "alert if more than 5 failed logins in 10 minutes").
Spark StreamingKnow the concept

Spark's real-time mode. Processes data in micro-batches (every few seconds) rather than true event-by-event like Flink. Slightly more latency, uses the same Spark engine.

SF equivalent: Scheduled Flow running every 5 minutes vs. a Platform Event-Triggered Flow that fires instantly. Spark Streaming is the scheduled approach — good enough for most use cases.
Processing — transforming data at scale
SparkLearn first

The workhorse. Distributed compute engine that runs SQL, Python, and ML jobs across a cluster. 10TB of data? Spark splits the work across 100 machines and combines results.

SF equivalent: SOQL + Apex Batch, but for the entire world's data. A SOQL query runs against your org. A Spark query runs against a data lake. You can write Spark in SQL (like SOQL) or Python (PySpark). Same mental model: query → transform → output.
ImpalaKnow the concept

Interactive SQL engine. Runs fast queries on data in HDFS or Kudu. Built for dashboards and ad-hoc analysis where you need answers in seconds.

SF equivalent: SOQL in Developer Console. Write a query, get results fast. Impala is the same experience but against a data lake. Analysts use it to power dashboards — just like Salesforce Reports, but on petabytes.
Storage — where data lives
HDFSLearn first

Hadoop Distributed File System. Stores massive files by splitting them across machines. Each chunk replicated 3x so nothing is lost. The foundation everything reads from.

SF equivalent: Salesforce Files / ContentVersion for the entire enterprise. You upload a file, it's stored and accessible. HDFS does the same for terabytes of raw data. You don't query HDFS directly — you use Spark or Impala, just like you use SOQL to read from objects.
KuduKnow the concept

Fast columnar storage for structured, mutable data. The sweet spot between HDFS (batch reads) and HBase (fast lookups). Great for dashboards on changing data.

SF equivalent: Big Objects that are actually fast. Stores massive structured data you can both update quickly AND run analytics on.
HBaseNice to have

NoSQL key-value store on HDFS. Billions of rows, millisecond lookups by key. Not for analytics — for fast "give me this one record" access.

SF equivalent: External Objects via OData. Massive external dataset, look up individual records by ID. Fast single-record retrieval, not reporting.
Iceberg / HiveKnow the concept

Table formats that organize lake files into queryable tables with schemas. Iceberg is the modern standard — time travel, schema evolution, ACID transactions.

SF equivalent: Custom Objects + Field History Tracking. Defines the schema (fields, types) for raw data. Time travel ≈ Field History — see what data looked like at any previous point.
Governance — security, lineage, compliance
AtlasKnow the concept

Data catalog + lineage tracker. Shows where every piece of data came from, every transformation, who touched it. Essential for compliance and auditing.

SF equivalent: Setup Audit Trail + Data Classification. Tracks who changed what. Atlas does the same for data — "this revenue number came from this Kafka topic → through this Spark job → into this table." Auditors ask this constantly.
RangerKnow the concept

Centralized access control. Who can see which data, down to the column and row level. One policy engine across all tools.

SF equivalent: Profiles + Permission Sets + Field-Level Security for the entire data platform. One policy: "marketing can see names but not SSNs" — enforced in Spark, Impala, NiFi, everywhere.
AI / ML
Cloudera ML (CML)Learn first

Cloudera's ML platform. Jupyter notebooks, model training on cluster resources, one-click deployment as REST APIs, monitoring for drift. Runs on Kubernetes.

SF equivalent: Einstein AI Platform + Prompt Builder. Einstein trains models on CRM data. CML does the same on any data — images, text, sensor data, whatever lives in the lake. Your AIRFX experience translates directly here.
Infrastructure
YARNNice to have

Resource manager. Decides which jobs get how much CPU and memory on the cluster. The traffic cop.

SF equivalent: Governor Limits + Queueable Apex. Salesforce limits CPU time per transaction. YARN does the same for a cluster — making sure one giant job doesn't starve everyone else.
LinuxKnow the concept

The OS everything runs on. Check logs, monitor resources, restart services, navigate files. CLI fluency, not sysadmin depth.

SF equivalent: You use bash daily with Claude Code. The gap is ops commands: checking disk space (df -h), reading logs (tail -f), restarting services (systemctl). Same terminal, different commands.

How data flows through the stack

Three real-world patterns. Each maps to a Salesforce pattern you already know.

Pattern 1: Batch ETL

Load, transform, and analyze large datasets on a schedule — like a nightly data sync.

1
NiFi
Pulls CSV/JSON from FTP, S3, APIs on a schedule
≈ Scheduled Flow pulling data from external system
2
HDFS (raw zone)
Raw files land untouched, as-is — nothing modified yet
≈ Staging custom object holding raw imported data
3
Spark
Cleans, joins, aggregates. Bad rows filtered, schemas enforced
≈ Apex Batch job transforming staging → final objects
4
Iceberg table
Clean data in queryable tables with versioning
≈ Custom Objects with Field History Tracking
5
Impala
Analysts query tables for dashboards and reports
≈ SOQL powering Salesforce Reports & Dashboards

Pattern 2: Real-time streaming

Process events as they happen — fraud detection, live telemetry. This is what Cloudera sells hardest.

1
Source
IoT sensors, app logs, clickstreams, transactions
≈ Inbound API calls hitting your Salesforce org
2
Kafka
Events published to topics. Decouples producers from consumers
≈ Platform Events published to event bus
3
Flink
Processes each event in real time — filter, enrich, detect anomalies
≈ Platform Event-Triggered Flow that fires instantly
4
Kudu
Fast-updating storage for dashboards on live data
≈ Custom Object updated in real time, powering a live dashboard
5
Alert / API
Trigger an action — email, API call, block transaction
≈ Outbound Message or Flow email alert

Pattern 3: ML pipeline

Train a model on historical data, deploy it, serve predictions — like Einstein but on any data.

1
Data lake
Historical data in HDFS / Iceberg tables
≈ All the records in your Salesforce objects
2
Spark
Feature engineering — clean, join, create training features
≈ Formula fields + roll-up summaries preparing data for Einstein
3
CML
Train model in Jupyter notebook, experiment, tune
≈ Einstein Model Builder training on your data
4
CML Deploy
One-click deploy as REST API endpoint
≈ Einstein Prediction deployed to a Record Page
5
App / API
App calls model API, gets prediction, takes action
≈ Next Best Action component on a Lightning page

Governance: the layer that wraps everything

Atlas + Ranger aren't a step in the pipeline — they're the security and audit layer across ALL steps.

✓
Ranger
Every query, every tool checks: "does this user have permission?"
≈ Profile + FLS checks on every SOQL query and page load
✓
Atlas
Every transformation logged: where data came from, who touched it
≈ Setup Audit Trail + Field History for your entire data estate
✓
Compliance
Auditors can trace any number to its source. Required for HIPAA, SOX, GDPR
≈ Your AIRFX compliance engine — same problem, platform-wide scope
