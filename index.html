The Cloudera stack,
in Salesforce terms.

Every tool mapped to something you already know. The goal isn't memorization — it's pattern recognition.

Ingestion — getting data in

NiFi learn first

Visual drag-and-drop data routing tool. You draw a pipeline on a canvas — "grab this file from an FTP server, transform the JSON, push it to Kafka." No code. All config.

SF equivalent: Flow Builder. You drag processors onto a canvas and connect them, just like you drag Flow elements. Each processor does one thing — route, filter, transform, deliver. If you can build a Screen Flow, you can understand NiFi.

Kafka learn first

A real-time message bus. Systems publish events to "topics" and other systems subscribe to read them. Nothing gets lost. Everything is ordered. Massively scalable.

SF equivalent: Platform Events. When you publish a Platform Event in Salesforce, any subscriber (Flow, Trigger, external system) picks it up. Kafka is that same concept but handling millions of events per second across an entire enterprise — not just within one Salesforce org.
Stream processing — reacting in real time

Flink know the concept

Processes data the instant it arrives in Kafka. Filters, aggregates, enriches, and routes events in real time — no waiting for a batch job.

SF equivalent: Platform Event-Triggered Flow. When a Platform Event fires, your Flow runs immediately on that event. Flink is the same idea: an event hits Kafka → Flink processes it in real time → output goes downstream. The difference is scale and complexity. Flink handles millions of events/sec with stateful logic (e.g., "alert if more than 5 failed logins in 10 minutes").

Spark Streaming know the concept

Spark's real-time mode. Processes data in micro-batches (every few seconds) rather than true event-by-event like Flink. Slightly more latency, but uses the same Spark engine teams already know.

SF equivalent: Scheduled Flow that runs every 5 minutes vs. a Platform Event-Triggered Flow that fires instantly. Both react to new data, but one has a small delay. Spark Streaming is the scheduled approach — good enough for most use cases.
Processing — transforming data at scale

Spark learn first

The workhorse. Distributed compute engine that runs SQL, Python, and ML jobs across a cluster of machines. If you have 10TB of data, Spark splits the work across 100 machines and combines the results.

SF equivalent: Think of it like SOQL + Apex Batch, but for the entire world's data. A SOQL query runs on Salesforce's servers against your org's data. A Spark query runs on a cluster against a data lake. You can write Spark queries in SQL (just like SOQL) or Python (PySpark). The mental model is the same: query → transform → output.

Impala know the concept

Interactive SQL engine. Runs fast queries on data in HDFS or Kudu. Built for BI dashboards and ad-hoc analysis where you need answers in seconds, not minutes.

SF equivalent: SOQL in the Developer Console. You write a query, you get results fast. Impala is the same experience but against a data lake instead of a Salesforce org. Analysts use it to power dashboards — same as Salesforce Reports, but on petabytes.
Storage — where data lives

HDFS learn first

Hadoop Distributed File System. Stores massive files by splitting them across machines. Each chunk is replicated 3x so nothing is ever lost. The foundation layer everything else reads from.

SF equivalent: Salesforce Files / ContentVersion, but for the entire enterprise. In Salesforce, you upload a file and it's stored and accessible. HDFS does the same but for terabytes/petabytes of raw data — logs, CSVs, images, JSON. You don't query HDFS directly — you use Spark or Impala to read from it, just like you use SOQL to read from Salesforce objects.

Kudu know the concept

Fast columnar storage for structured, mutable data. Sits between HDFS (great for batch reads, bad for updates) and HBase (great for lookups, bad for analytics). The sweet spot for dashboards on changing data.

SF equivalent: Big Objects. Salesforce Big Objects store massive structured datasets you can query but rarely update. Kudu is similar but allows fast updates AND fast analytics — like if Big Objects were actually fast and you could run Reports on them.

HBase nice to have

NoSQL key-value store on top of HDFS. Billions of rows, millisecond lookups by key. Not for analytics — for fast "give me this one record" access.

SF equivalent: External Objects via OData. You have a massive external dataset and you look up individual records by ID. HBase is the storage engine that makes that lookup fast. You wouldn't run a Report on it — you'd query one record at a time.

Iceberg / Hive know the concept

Table formats that organize files in HDFS into queryable tables with schemas. Iceberg is the modern standard — supports time travel (query data as it existed last Tuesday), schema evolution, and ACID transactions.

SF equivalent: Custom Objects + Field History Tracking. When you create a Custom Object, you define the schema (fields, types). Iceberg does the same for data lake files. Field History Tracking ≈ Iceberg's time travel — both let you see what the data looked like at a previous point.
Governance — security, lineage, compliance

Atlas know the concept

Data catalog and lineage tracker. Shows where every piece of data came from, every transformation it went through, and who touched it. Essential for compliance and auditing.

SF equivalent: Setup Audit Trail + Data Classification. Setup Audit Trail tracks who changed what config. Atlas does the same but for data — "this revenue number in the dashboard came from this Kafka topic → through this Spark job → into this Kudu table." In regulated industries, auditors ask this question constantly.

Ranger know the concept

Centralized access control. Defines who can see which data, down to the column and row level. One policy engine across all tools — Spark, Impala, HDFS, Kafka, everything.

SF equivalent: Profiles + Permission Sets + Field-Level Security, but for the entire data platform. In Salesforce, you control who sees which fields on which objects. Ranger does the same across every tool in the cluster. One policy: "the marketing team can see customer names but not SSNs" — enforced in Spark, Impala, NiFi, everywhere.
AI / ML — Cloudera machine learning

Cloudera ML (CML) learn first

Cloudera's ML platform. Jupyter notebooks for experimentation, model training on cluster resources, one-click deployment as REST APIs, and monitoring for model drift. Runs on Kubernetes.

SF equivalent: Einstein AI Platform + Prompt Builder. Einstein trains models on your CRM data and deploys predictions. CML does the same but on any data — images, text, sensor data, whatever lives in the data lake. Your AIRFX experience translates directly: you built AI that makes decisions on enterprise data. CML is the platform that hosts that kind of work at Cloudera.
Infrastructure

YARN nice to have

Resource manager for the cluster. Decides which jobs get how much CPU and memory. When you submit a Spark job, YARN allocates containers on cluster nodes to run it.

SF equivalent: Governor Limits + Queueable Apex. Salesforce limits how much CPU time and heap each transaction gets. YARN does the same for a cluster — making sure one giant Spark job doesn't eat all the resources and starve everyone else. It's the traffic cop.

Linux know the concept

The OS everything runs on. You need to be comfortable in a terminal: check logs (tail -f), monitor resources (top, df -h), restart services (systemctl), navigate files (cd, ls, cat).

SF equivalent: You already use bash every day with Claude Code. The gap is ops-specific commands: checking disk space, reading service logs, managing permissions, restarting services. It's not a different world — just a different set of commands in the same terminal you already use.
